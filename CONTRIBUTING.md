# Contributing to AI Product Development Skills

Thank you for your interest in contributing! This project aims to help product teams create better documentation through prescriptive frameworks.

## Ways to Contribute

### 1. Share Your Experience
Open an issue to share:
- How you're using these skills
- What worked well
- What could be improved
- Results you've achieved

### 2. Submit Example PRDs or User Stories
We welcome real-world examples! Submit PRs with:
- Anonymized PRDs or user stories
- Scoring breakdown (what scores 24/24 or 18/18)
- Context about your domain/industry
- Lessons learned

**Example format:**
```markdown
## Example: [Domain] - [Feature Type]

[Your PRD or User Story]

### Scoring
- Section 1: 3/3 - [Why]
- Section 2: 2/3 - [What could improve]
...

### What Worked Well
- [Insight 1]
- [Insight 2]

### Challenges
- [Challenge 1 and how you addressed it]
```

### 3. Improve Documentation
Found unclear guidance? Submit PRs to improve:
- Clarify ambiguous requirements
- Add more ❌/✅ examples
- Improve rubric descriptions
- Fix typos or formatting

### 4. Suggest Framework Enhancements
Open issues to propose:
- New sections for PRD template
- Additional user story elements
- Industry-specific adaptations
- Scoring rubric adjustments

### 5. Report Issues
Found a problem? Open an issue with:
- What you were trying to do
- What happened vs. what you expected
- Steps to reproduce (if applicable)
- Relevant context (domain, team size, etc.)

## Guidelines

### For Example Submissions
- **Anonymize**: Remove company names, product details, customer data
- **Quality**: Examples should score well (20+/24 for PRDs, 15+/18 for stories)
- **Format**: Follow the template structures exactly
- **Context**: Explain what makes the example notable

### For Documentation Improvements
- **Clarity**: Use simple, direct language
- **Examples**: Show don't tell (❌/✅ comparisons)
- **Consistency**: Match existing tone and structure
- **Testing**: Verify examples actually work

### For Framework Changes
- **Rationale**: Explain the problem you're solving
- **Evidence**: Share data or experience supporting the change
- **Backwards Compatible**: Consider impact on existing users
- **Discussion First**: Open an issue before major PRs

## Pull Request Process

1. **Fork the repository**
2. **Create a feature branch** (`git checkout -b feature/your-feature`)
3. **Make your changes**
4. **Test thoroughly** (if adding examples, score them)
5. **Commit with clear messages** (`git commit -m "Add: example PRD for SaaS onboarding"`)
6. **Push to your fork** (`git push origin feature/your-feature`)
7. **Open a Pull Request** with description of changes

### PR Description Template
```markdown
## What
[Brief description of changes]

## Why
[Problem being solved or improvement being made]

## Testing
[How you verified the changes work]

## Related Issues
[Link to related issues if any]
```

## Code of Conduct

### Our Standards
- **Be respectful**: Treat all contributors with respect
- **Be constructive**: Provide helpful feedback
- **Be collaborative**: Work together to improve the project
- **Be patient**: Remember everyone is learning

### Unacceptable Behavior
- Harassment or discrimination
- Trolling or insulting comments
- Publishing others' private information
- Other conduct inappropriate in a professional setting

## Questions?

- **General questions**: Open a [Discussion](../../discussions)
- **Bug reports**: Open an [Issue](../../issues)
- **Feature requests**: Open an [Issue](../../issues) with `enhancement` label
